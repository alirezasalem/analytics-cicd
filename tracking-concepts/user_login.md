# user_login

Tracks when a user successfully completes the login process and authenticates into the application.

---

## Overview

This event fires when a user successfully logs into their account. It captures the authentication method used (e.g., email/password, Google SSO, Apple ID) and associates the session with the user's unique identifier. This data helps the business understand login patterns, authentication method preferences, and user engagement across sessions.

---

## Trigger Rules

- ✅ The user must have successfully completed the authentication flow (API returns success, session is established)
- ✅ The `user_id` must be available and populated in the dataLayer before this event fires
- ✅ Fire only **after** login completion is confirmed server-side — do not fire on form submission or button click
- ❌ Do NOT fire on failed login attempts
- ❌ Do NOT fire on session restoration or "remember me" auto-logins (only explicit user-initiated logins)
- ❌ Do NOT fire on account creation — use a separate registration event

## Business Rules & Edge Cases

1. **Timing is critical**: The `user_id` must be set in the dataLayer *before* this event fires. Coordinate with your auth callback to ensure the user object is fully hydrated.

2. **Do not fire on page load for returning sessions**: If a user returns with an active session cookie, this event should NOT fire. It tracks only explicit login actions.

3. **Standardize `login_method` values**: Use a consistent taxonomy across your codebase. Recommended values:
   - `email_password`
   - `google_sso`
   - `apple_id`
   - `facebook`
   - `magic_link`
   - `saml_sso`

4. **SPA considerations**: In single-page applications, ensure the event fires after the login modal closes and the auth state is confirmed, not on form submission.

5. **PII restriction**: The `user_id` should be a hashed or anonymized identifier — never pass raw email addresses or PII in this field.

6. **Deduplication**: If your app has multiple login entry points (modal, dedicated page, checkout flow), ensure only one `user_login` event fires per successful authentication.
