# purchase

Tracks a completed transaction when a customer successfully finishes the checkout process.

---

## Overview

The `purchase` event fires when a customer lands on the order confirmation page after completing a transaction. This event is critical for revenue attribution, marketing ROI analysis, and understanding customer purchasing behavior. It captures the full transaction details including order value, tax, shipping, applied discounts, and all purchased items.

---

## Trigger Rules

- **Must** fire on page load of the `/checkout/confirmation` page
- **Must** only fire after the order has been successfully processed by the backend (confirmation page render implies success)
- **Must** include a unique `transaction_id` — GA4 uses this for server-side deduplication of duplicate purchase events
- **Must not** fire on checkout error pages or payment failure states
- **Must not** fire if the user refreshes the confirmation page and the transaction has already been recorded (implement client-side deduplication if confirmation page is reloadable)
- **Must not** fire during browser back-button navigation to the confirmation page

## Business Rules & Edge Cases

1. **Transaction ID deduplication**: The GA4 tag in GTM must have transaction deduplication enabled. Use a consistent, unique `transaction_id` format that will not repeat across orders. If a user refreshes the confirmation page, the same `transaction_id` will be sent again — GA4 will deduplicate server-side, but implement client-side guards (e.g., sessionStorage flag) to avoid unnecessary hits.

2. **Items array completeness**: Every item in the `items` array must include `item_id`, `item_name`, `price`, `quantity`, and `item_category`. Missing fields will result in incomplete product-level reporting in GA4.

3. **Currency consistency**: The `currency` value must match the currency used for `value`, `tax`, `shipping`, and all item `price` values. Do not mix currencies within a single purchase event.

4. **Value calculation**: The `value` parameter should represent the total amount charged to the customer, typically: `(sum of item prices × quantities) + tax + shipping - discounts`.

5. **Coupon codes**: If no coupon was applied, omit the `coupon` parameter entirely rather than sending an empty string.

6. **SPA considerations**: If the checkout flow is a single-page application, ensure the purchase event only fires once when the order is confirmed — not on virtual page view changes or route transitions.

7. **PII restrictions**: The `user_id` must be an anonymized or hashed identifier. Do not send email addresses, full names, or other personally identifiable information.
