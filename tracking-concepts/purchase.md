# purchase

Tracks a completed e-commerce transaction when a customer successfully finishes the checkout process.

## Overview

This event fires when a customer lands on the order confirmation page after completing a purchase. It captures the full transaction details including order total, tax, shipping, and all purchased items. This is the most critical e-commerce event for revenue attribution, conversion tracking, and understanding purchase behavior.

## Trigger Rules

- The user must land on the `/checkout/confirmation` page
- The event fires on **page load**, not on any user click action
- A valid `transaction_id` must be present (required for deduplication)
- The order data must be fully resolved before the push (wait for backend confirmation data to populate the page)
- Do NOT fire this event on page refresh if the transaction was already recorded—implement client-side or server-side deduplication
- Do NOT fire if the user navigates to the confirmation page without completing a valid transaction (e.g., via direct URL access with no order context)

## Business Rules & Edge Cases

1. **Transaction ID Deduplication**: GA4 tag must have transaction_id deduplication enabled to prevent double-counting on page refreshes. Implement additional client-side protection (e.g., sessionStorage flag) for belt-and-suspenders safety.

2. **Items Array Requirements**: Every item in the `items` array must include `item_id`, `item_name`, `price`, `quantity`, and `item_category`. Missing required item properties will result in incomplete product-level reporting.

3. **Currency Formatting**: The `value`, `tax`, `shipping`, and `items[].price` must be numeric values (not strings). Currency should be a valid ISO 4217 code. All monetary values in a single purchase must use the same currency.

4. **Coupon Handling**: If no coupon was applied, omit the `coupon` parameter entirely rather than passing an empty string or null.

5. **PII Restrictions**: The `user_id` must be a hashed or anonymized identifier. Never pass email addresses, phone numbers, or other personally identifiable information.

6. **SPA Considerations**: For single-page applications, ensure this event only fires once per actual completed transaction, even if the user navigates away and returns to the confirmation view.
