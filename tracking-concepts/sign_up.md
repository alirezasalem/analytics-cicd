# sign_up

Tracks when a user successfully creates a new account on the platform.

---

## Overview

The `sign_up` event fires when a user completes the account registration process. This event is critical for measuring user acquisition, understanding which signup methods drive the most registrations, and calculating conversion rates from visitor to registered user. It should only fire once per successful account creation, not on form submission attempts that fail validation.

---

## Trigger Rules

- **Must fire when:** The account creation API returns a success response (user record created in backend)
- **User action:** Form submission of the signup/registration form
- **Precondition:** The `user_id` must be available in the dataLayer before this event fires (wait for backend confirmation)
- **Must NOT fire on:**
  - Failed form validation (client-side errors)
  - Failed API response (duplicate email, server error, etc.)
  - Page refresh after signup (deduplicate using session storage or similar)
  - Users who are already logged in attempting to access signup

## Business Rules & Edge Cases

1. **Timing dependency:** Do NOT fire this event on form submit click. Wait for the backend API to confirm successful account creation and return the `user_id`. This prevents tracking failed signups.

2. **user_id availability:** The GTM trigger should include a condition that verifies `user_id` is defined and non-empty in the dataLayer before the tag fires.

3. **Deduplication:** Implement client-side deduplication to prevent duplicate events if the user refreshes the confirmation page. Use `sessionStorage` to track that signup was already fired for this session.

4. **Sign-up method values:** Use consistent, lowercase values for `sign_up_method`. Standardize across the codebase (e.g., always `google`, never `Google` or `google_oauth`).

5. **SPA handling:** For single-page applications, ensure the event fires only once per registration flow, even if the user navigates away and returns to a confirmation screen.

6. **PII restriction:** The `user_id` must be a hashed or anonymized identifier—never pass email addresses, names, or other personally identifiable information in this field.
