# sign_up

Tracks when a new user successfully creates an account on the platform.

---

## Overview

This event fires after a user completes the registration process and the backend confirms the account was created. It captures the authentication method used (email, Google, Facebook, etc.) and the newly assigned user ID. This data helps measure signup funnel conversion rates, identify the most effective registration methods, and establish the user's identity for downstream behavioral analysis.

---

## Trigger Rules

- The user must be on the `/signup` page (or a signup modal/flow)
- The signup form submission must result in a **successful** account creation (HTTP 201 Created response from the backend)
- The event must fire **only after** the backend API confirms the account was created — never on form submit or button click alone
- Do **not** fire if:
  - The API returns a validation error (invalid email, weak password, etc.)
  - The API returns an "account already exists" error
  - The user abandons the signup flow before completion
  - The signup fails for any reason (network error, server error, etc.)

## Business Rules & Edge Cases

1. **Wait for backend confirmation**: The event must only fire after receiving a successful 201 Created response from the account creation API. Do not fire optimistically on button click.

2. **Duplicate prevention**: If a user attempts to sign up with an email that already exists, the backend will return an error — do not fire the event in this scenario.

3. **Method values should be consistent**: Use lowercase, standardized values for the `method` parameter (e.g., `email`, `google`, `facebook`, `apple`, `sso`). Coordinate with backend to ensure consistent naming.

4. **User ID availability**: The `user_id` must come from the API response. If for any reason the user ID is not returned, log an error and do not fire the event with a missing or placeholder ID.

5. **SPA considerations**: If the signup flow is a multi-step wizard or modal that doesn't trigger a page reload, ensure the event fires at the correct step (final success confirmation), not on intermediate steps.
