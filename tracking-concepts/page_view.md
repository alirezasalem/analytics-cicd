# page_view

Tracks every page view across the site, including both traditional page loads and client-side navigation in single-page applications (SPAs).

---

## Overview

The `page_view` event fires whenever a user views a new page. This includes the initial page load when a user first arrives, as well as any subsequent navigation that changes the URL without a full page reload (common in React, Vue, or Angular apps). This event is foundational for understanding user journeys, identifying high-traffic pages, and measuring content performance.

---

## Trigger Rules

- **Must fire on initial page load** — when the DOM is fully ready on first arrival
- **Must fire on SPA navigation** — whenever the browser history changes (e.g., user clicks an internal link that updates the URL via `pushState` or `replaceState`)
- **Must fire once per distinct page view** — do not double-fire if both DOM Ready and History Change occur for the same URL
- **Must NOT fire on hash-only changes** (e.g., `#section1` to `#section2`) unless these represent distinct content views in your SPA routing
- **Must NOT include PII in `page_location`** — strip or mask any query parameters containing email addresses, names, or other personally identifiable information before pushing to the dataLayer

## Business Rules & Edge Cases

1. **PII Filtering** — Before pushing `page_location`, scan query parameters for PII (email, phone, SSN patterns) and remove or redact them. Example: `?email=john@example.com` should become `?email=[REDACTED]` or be stripped entirely.

2. **SPA Deduplication** — In SPAs, both a DOM Ready trigger and a History Change trigger may fire in quick succession. Implement logic to ensure only one `page_view` fires per unique URL. A common pattern is to track the last-fired URL and skip if it matches.

3. **GTM Trigger Configuration** — This event should fire on two GTM triggers: **DOM Ready** (for initial page load) and **History Change** (for SPA navigation). Both triggers should point to the same tag.

4. **Empty Referrer Handling** — `page_referrer` will be empty or undefined for direct traffic, bookmarks, or when referrer headers are blocked. This is expected; do not substitute a placeholder value.

5. **User ID for Anonymous Users** — If the user is not logged in, either omit `user_id` from the push entirely or set it to `undefined`. Do not send empty strings or placeholder values like `"guest"`.

6. **Device Type Detection** — Determine `device_type` using viewport width or user-agent parsing at the time of the page view. Use consistent breakpoints across your codebase (e.g., mobile < 768px, tablet 768–1024px, desktop > 1024px).
