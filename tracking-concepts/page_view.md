# page_view

Tracks when a user views any page on the website, including both traditional page loads and single-page application (SPA) route changes.

---

## Overview

The `page_view` event fires every time a user lands on a page or navigates to a new view within the application. This is the foundational event for understanding traffic patterns, user journeys, and content engagement across the site. It captures page context (type and category) along with the user's authentication state, enabling segmentation of pageviews by logged-in versus anonymous visitors.

---

## Trigger Rules

- **Must fire** on the initial browser page load (traditional navigation or direct URL entry)
- **Must fire** on client-side route changes in single-page applications (SPA) when the URL or view changes without a full page reload
- **Must fire** once per distinct page view â€” do not double-fire on the same route
- **Must NOT fire** on hash-only changes (e.g., `#section`) unless they represent meaningful view changes defined by the application
- **Must NOT fire** on background tab preloads or bot traffic (if detectable)
- User authentication state (`user_login_status`, `user_id`) must be resolved before the event fires

## Business Rules & Edge Cases

1. **PII Protection (Critical):** The `user_id` must be hashed or anonymized before transmission. Never send email addresses, plain-text usernames, or other personally identifiable information. This is required for GDPR/CCPA compliance.

2. **SPA Navigation Handling:** In GTM, configure a History Change trigger to capture client-side route changes in addition to the standard Page View trigger for initial loads. Both must push the same event structure.

3. **user_id for Logged-Out Users:** If the user is not authenticated, either omit `user_id` entirely or send an empty string (`""`). Do not send placeholder values like `"guest"` or `"anonymous"`.

4. **Timing Dependency:** Ensure all dataLayer variables (`page_type`, `page_category`, `user_login_status`, `user_id`) are populated before the `page_view` push. For SPAs, wait for route resolution and any async user-state checks to complete.

5. **Deduplication:** Implement logic to prevent duplicate `page_view` events on rapid successive route changes (e.g., user clicking back/forward quickly). A minimum debounce of 100â€“200ms between fires is recommended.

6. **Fallback Values:** If `page_type` or `page_category` cannot be determined, use `"unknown"` rather than omitting the parameter, to maintain data consistency in reports.
