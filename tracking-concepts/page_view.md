# page_view

Tracks when a user loads any page in the application, capturing the page type and category for content engagement analysis.

---

## Overview

The `page_view` event fires every time a page finishes loading in the user's browser. This is the foundational event for understanding user navigation patterns, content consumption, and site structure performance. It provides the baseline data needed for page-level reporting, funnel analysis, and identifying which content types drive the most engagement.

---

## Trigger Rules

- **Must fire** when the DOM is fully ready (DOM Ready state), not on initial page request
- **Must fire** on every page load, including:
  - Initial page loads from direct navigation or external links
  - Full page refreshes
  - SPA route changes that represent a new "page" view
- **Must ensure** all dataLayer variables (`user_id`, `page_type`, `page_category`) are populated before the event fires
- **Must NOT fire** multiple times for the same page view (e.g., avoid duplicate triggers from lazy-loaded content)
- **Must NOT fire** on hash-only URL changes unless they represent a distinct content view

## Business Rules & Edge Cases

1. **Timing dependency**: The dataLayer variables (`user_id`, `page_type`, `page_category`) must be set before this event fires. In GTM, configure the trigger to fire on DOM Ready, and ensure any variable-setting scripts run earlier (e.g., in the `<head>` or via a higher-priority tag).

2. **SPA handling**: For single-page applications, fire this event on each virtual page view when the route changes. Use a History Change trigger or custom event to detect navigation, and update `page_type` and `page_category` before pushing the event.

3. **Anonymous users**: When `user_id` is not available (logged-out users), either omit the parameter entirely or set it to `undefined`. Do not pass empty strings or placeholder values.

4. **Page type consistency**: Maintain a controlled vocabulary for `page_type` values across the codebase. Document all valid values to ensure consistent reporting (e.g., always use "product_detail" not "product-detail" or "PDP").

5. **Initial page load vs. refresh**: This event should fire identically whether the user arrives via navigation or refreshes the page. Ensure no conditional logic skips the event on refresh.
