# page_view

Tracks when a user views a page, whether on initial site load or during single-page application (SPA) navigation.

---

## Overview

The `page_view` event fires every time a user lands on a new page or navigates within a single-page application. This is a foundational event for understanding user journeys, traffic sources, and content engagement. It captures the current URL, the referring page, and basic user/device context to support audience segmentation and attribution analysis.

---

## Trigger Rules

- **Must fire** on initial page load when the DOM is ready
- **Must fire** on SPA route changes (History API state changes)
- **Must fire** for every distinct page path, including query parameters
- **Must NOT fire** multiple times for the same page view (e.g., don't re-fire on scroll or minor DOM updates)
- **Must NOT include** personally identifiable information (PII) in the `page_location` parameter—strip email addresses, tokens, or other sensitive query parameters before pushing

## Business Rules & Edge Cases

1. **PII Filtering:** Before pushing `page_location`, strip any query parameters that may contain PII (e.g., `?email=`, `?token=`, `?reset_code=`). This is a compliance requirement.

2. **SPA Navigation:** For single-page applications, fire this event on every History API change (`pushState`, `replaceState`, or `popstate` event). Do not wait for a full page reload.

3. **Referrer on Initial Load:** On the first page load, `page_referrer` should reflect `document.referrer`. On subsequent SPA navigations, it should reflect the previous in-app page URL.

4. **User ID Handling:** If the user is not logged in, either omit `user_id` entirely or set it to `undefined`. Do not pass empty strings or placeholder values.

5. **Device Type Detection:** Determine `device_type` using viewport width or user-agent parsing. Ensure consistent logic across the codebase (e.g., mobile ≤ 767px, tablet 768–1024px, desktop > 1024px).

6. **Deduplication:** Ensure the event does not fire twice for the same page view. This can happen if both a DOM Ready trigger and a History Change trigger fire on initial load—guard against this in your implementation.
