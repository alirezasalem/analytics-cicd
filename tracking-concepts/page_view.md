# page_view

Tracks every page view across the site, including initial page loads and single-page application (SPA) route changes.

---

## Overview

The `page_view` event fires whenever a user lands on a page or navigates to a new view within a single-page application. This is the foundational event for understanding traffic patterns, user journeys, and content engagement. It captures the URL, referring page, page title, and device information to enable audience segmentation and attribution analysis.

---

## Trigger Rules

- **Must fire** on initial DOM Ready when any page loads
- **Must fire** on SPA History Change events (pushState/popstate) when the route changes
- **Must fire** for every unique page path visited, including query parameter variations if they represent distinct content
- **Must NOT fire** multiple times for the same page view during a single navigation action (deduplicate rapid history changes)
- **Must NOT include** personally identifiable information (PII) in the `page_location` parameter — apply PII filtering before pushing to dataLayer

## Business Rules & Edge Cases

1. **PII Filtering Required**: Before pushing `page_location`, strip any PII from the URL. This includes email addresses in query parameters, tokens, or any user-specific identifiers that could appear in the URL path or query string.

2. **SPA Navigation Handling**: This event must fire on History Change triggers, not just initial page loads. Ensure your GTM configuration includes both DOM Ready and History Change as triggers for this tag.

3. **User ID Availability**: If the user is not logged in, either omit `user_id` entirely or set it to `null`. Do not pass placeholder values like `"anonymous"` or empty strings.

4. **Referrer Edge Cases**: On direct traffic or when the referrer is unavailable (e.g., HTTPS to HTTP transitions), `page_referrer` may be an empty string. This is expected behavior — do not fabricate a referrer value.

5. **Deduplication on Rapid Navigation**: In SPAs with rapid route changes (e.g., redirect chains), ensure the event fires only once per final destination page, not for intermediate routes that flash briefly.

6. **Query Parameter Handling**: Decide with your analytics team whether query parameters should be included in `page_location`. If campaign parameters (utm_*) are present, preserve them; if sensitive parameters are present, strip them per the PII filtering rule.
