# generate_lead

Tracks successful contact form submissions after server-side validation confirms the lead was captured.

---

## Overview

This event fires when a user completes and submits the contact form on the `/contact` page, and the server responds with a successful acknowledgment. It captures metadata about which form was submitted and the type of lead, enabling the marketing team to measure lead generation performance and attribute conversions by form variant and lead category. This event is critical for understanding which contact touchpoints drive business inquiries.

---

## Trigger Rules

- ✅ User is on the `/contact` page
- ✅ User has clicked the form submit button AND the form has passed client-side validation
- ✅ The backend API returns a success response (HTTP 200 or 201)
- ✅ The dataLayer push occurs only after the successful API response is received
- ❌ Do NOT fire on button click alone—the submit action must be confirmed by the server
- ❌ Do NOT fire on form validation errors or failed submissions
- ❌ Do NOT fire if the API returns an error status (4xx, 5xx)

## Business Rules & Edge Cases

1. **Fire after API success only**: The event must fire inside the success callback of your form submission handler (e.g., `fetch().then()`, `axios` success, or XHR `onload` with status check). Never fire on the button click event itself.

2. **PII restrictions are critical**: Only capture form metadata (`form_id`, `form_name`, `lead_category`). Under no circumstances should you capture or pass user name, email address, phone number, message content, or any other personally identifiable information.

3. **Deduplication**: If your form submission handler could potentially fire multiple times (e.g., user double-clicks, retry logic), implement a flag to ensure `generate_lead` fires only once per successful submission.

4. **Error handling**: If the API returns an error response after an initial success (rare edge case with async processing), do not push a second event. The first successful response is the source of truth.

5. **SPA considerations**: If the contact page is part of a single-page application, ensure the dataLayer push fires correctly within the component lifecycle and that the form state resets appropriately after navigation.

6. **Form variant tracking**: Use distinct `form_id` values for A/B test variants, embedded forms vs. standalone pages, or different form versions to enable accurate conversion attribution.
