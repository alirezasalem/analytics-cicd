# add_to_cart

Tracks when a user successfully adds a product to their shopping cart.

## Overview

The `add_to_cart` event fires when a product has been successfully added to the user's shopping cart. This event is triggered **after** the cart API returns a successful response, not on button click. It helps the business understand product demand, identify popular items, and measure the effectiveness of product pages, search results, and recommendation modules in driving cart additions.

## Trigger Rules

- ✅ User initiates an "add to cart" action (button click, quantity increase, etc.)
- ✅ The cart API request completes with an HTTP 2xx success response
- ✅ The event fires **after** API success confirmation, not on the initial click
- ✅ Debouncing must be implemented to prevent duplicate events from rapid clicks or retry logic
- ❌ Do NOT fire if the API returns an error (4xx, 5xx, or timeout)
- ❌ Do NOT fire on click before API confirmation
- ❌ Do NOT fire duplicate events for the same add action (implement debounce)

## Business Rules & Edge Cases

1. **Fire on API success only**: The application layer must emit a custom event (or callback) after the cart API returns HTTP 2xx. GTM listens for this custom event, not the button click.

2. **Clear ecommerce object before push**: Always push `{ ecommerce: null }` immediately before the add_to_cart push to prevent data bleed from previous ecommerce events.

3. **Debouncing for duplicate prevention**: Implement debouncing logic to prevent multiple events if the user rapidly clicks "Add to Cart" or if retry logic fires multiple API calls. Only one event should fire per successful add action.

4. **Multiple surfaces**: This event fires from various page types (PDP, search results, quick-view modals, recommendation carousels). Ensure `item_list_id` and `item_list_name` accurately reflect where the user discovered the product.

5. **Value calculation**: The `value` parameter should equal `price × quantity` for the items being added in this specific action (after any discounts).

6. **Currency consistency**: Use the same currency code across all ecommerce events in a session. Do not mix currencies.

7. **Empty string vs. omission**: For optional string parameters with no value, pass an empty string (`""`) rather than omitting the parameter, to maintain consistent data structure.
