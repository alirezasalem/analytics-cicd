# view_item

Fires when a user views a product detail page and the full product data is rendered in the DOM.

---

## Overview

`view_item` records every visit to a product detail page. It tells the business which products attract attention and feeds into funnel analysis between browsing and purchasing. The event fires once per page render — including back-navigation and client-side route changes in the SPA — but only after the product data has fully loaded, not on the initial skeleton screen.

---

## Trigger Rules

- The user has navigated to a product detail page (URL pattern: `/products/:slug`)
- The product data API response has returned HTTP 200 and the product title, price, and SKU are present in the DOM
- This is **not** a redirect away from the page (e.g. out-of-stock redirect to category)
- The event fires **once per page view** — if the user refreshes or navigates back, it fires again; it does **not** fire again on tab focus or scroll
- If the page renders with a loading skeleton and data loads asynchronously, the push must be deferred until the data is available — do **not** fire on skeleton render
- SPA navigation (React Router / Nuxt `<NuxtLink>`) counts as a new page view and must fire the event

## Business Rules & Edge Cases

1. **Clear ecommerce before push.** Always call `dataLayer.push({ ecommerce: null })` immediately before the `view_item` push. GA4 / GTM merges ecommerce objects across pushes, which causes data contamination if items from a previous event are still present.
2. **Fire after API success, not on route change.** The event must not fire until the product API returns HTTP 200 and the title + price fields are rendered. Firing on skeleton render inflates view counts with failed product loads.
3. **SPA route changes count as page views.** Use your router's `afterEach` hook (Nuxt: `useRouter().afterEach`, React Router: `useEffect` on `location.pathname`) to detect navigation and re-fire the event.
4. **Do not send PII as `user_id`.** The value must be a hashed or opaque internal identifier. Email addresses, names, or phone numbers must never appear in this field.
5. **`item_id` must be stable.** The same physical product must always send the same `item_id`, regardless of variant selected. Variant information goes in `item_variant`, not `item_id`.
6. **`item_variant` is required when variants exist.** If the product has selectable options (size, colour) and none is selected yet, send the default visible variant. Do not omit the field if variants are defined in the product catalog.
7. **Currency must match the displayed price.** If the storefront shows localised prices, the `currency` field must reflect that — do not hardcode `"EUR"` if the user is seeing GBP.
