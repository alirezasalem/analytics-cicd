# login

Tracks when a user successfully authenticates into their account.

## Overview

This event fires immediately after a user completes a successful login. It captures the authentication method used (e.g., email/password, Google SSO, magic link) and the user's internal identifier. This data helps the business understand login patterns, identify friction in authentication flows, and measure the adoption of different sign-in methods.

## Trigger Rules

- ✅ The authentication API must return a successful response (do not fire on button click alone)
- ✅ The user must be on the `/login` page path (or initiated login from this flow)
- ✅ Fire immediately upon receiving the success response, before any redirect occurs
- ✅ Use transport beaconing to ensure the event is captured even if the page redirects quickly after login
- ❌ Do NOT fire if authentication fails or returns an error
- ❌ Do NOT fire on session refresh or "remember me" silent re-authentication
- ❌ Do NOT include cleartext email addresses, usernames, or other PII in the `user_id` field

## Business Rules & Edge Cases

1. **PII Restriction (Critical):** The `user_id` must be a hashed value or internal database identifier. Never pass cleartext emails, usernames, phone numbers, or any other personally identifiable information.

2. **Redirect Handling:** Login flows often redirect immediately after success. Configure GTM to use `navigator.sendBeacon` or equivalent transport method to ensure the event fires before the page unloads.

3. **Fire on API Success, Not Click:** The event must fire only after the backend confirms successful authentication. Do not fire optimistically on form submission or button click.

4. **Method Consistency:** Use consistent, lowercase values for the `method` parameter across all login implementations (e.g., `google_sso`, not `Google SSO` or `google-sso`).

5. **Single Sign-On Flows:** For SSO redirects (Google, Apple, SAML), fire the event on the callback page when authentication is confirmed, not on the initial redirect out.

6. **Session Restoration:** Do not fire this event when a returning user's session is automatically restored via cookies or tokens — only on explicit user-initiated logins.
