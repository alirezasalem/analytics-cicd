# login

Tracks when a user successfully authenticates into their account.

---

## Overview

This event fires immediately after a user successfully logs into the application. It captures the authentication method used (e.g., email/password, Google SSO, magic link) and the user's internal identifier. This data helps the business understand login patterns, identify preferred authentication methods, and measure authentication success rates across different sign-in flows.

---

## Trigger Rules

- ✅ The authentication API returns a successful response (200-level status code confirming valid credentials)
- ✅ The user is now in an authenticated session state
- ✅ The event fires **after** server confirmation, not on form submission or button click
- ✅ Use transport beaconing to ensure the event is sent even if the page immediately redirects after login
- ❌ Do NOT fire on failed login attempts
- ❌ Do NOT fire on session restoration (returning user with valid session cookie)
- ❌ Do NOT fire if the authentication response is pending or errored

## Business Rules & Edge Cases

1. **PII Restriction (Critical):** The `user_id` parameter must be a hashed value or internal database ID only. Never pass cleartext emails, usernames, phone numbers, or any other personally identifiable information. Violations may breach privacy regulations and GA4 terms of service.

2. **Timing:** Fire this event only after receiving a successful authentication response from the server. Do not fire optimistically on button click or form submission.

3. **Redirect Handling:** Many login flows immediately redirect to a dashboard or return URL. Use `navigator.sendBeacon()` or GTM's transport beacon feature to ensure the event is sent before the page unloads.

4. **Method Values:** Use consistent, lowercase, snake_case values for the `method` parameter. Common values include: `email_password`, `google_sso`, `facebook_sso`, `apple_id`, `magic_link`, `sms_otp`.

5. **Session Restoration:** This event should only fire for active login actions, not when a returning user's session is automatically restored via cookies or tokens.

6. **Selector Clarification Needed:** The form selector for the login trigger is not specified in the current spec. Work with the analytics team to define the exact DOM selector or custom event name used to identify successful logins in GTM.
