# page_view

Tracks when a user views any page on the website, including both initial page loads and single-page application (SPA) route changes.

---

## Overview

The `page_view` event fires every time a user lands on a page or navigates to a new view within the application. This is the foundational event for understanding traffic patterns, user journeys, and content performance. It captures page context (type and category) alongside user authentication state, enabling segmentation of behavior between logged-in and anonymous users.

---

## Trigger Rules

- **Must fire** on the initial browser page load (full page refresh or direct navigation)
- **Must fire** on client-side route changes in single-page applications (SPA navigation via History API)
- **Must fire** after the page content and dataLayer variables are fully available
- **Must NOT** fire multiple times for the same page view (deduplicate if multiple triggers overlap)
- **Must NOT** transmit unhashed user IDs — all `user_id` values must be hashed/anonymized before the push

## Business Rules & Edge Cases

1. **SPA Navigation Handling**: In single-page applications, use GTM's History Change trigger in addition to the standard Page View trigger to capture client-side route changes. Ensure the dataLayer is updated with new page context before firing.

2. **PII Compliance**: The `user_id` parameter is flagged as a PII risk. Before pushing to the dataLayer, the raw user ID **must** be hashed using SHA-256 (or equivalent one-way hash) to comply with GDPR and CCPA requirements. Never transmit plain-text email addresses, account numbers, or other identifiable information.

3. **Logged-Out Users**: When `user_login_status` is `"logged_out"`, set `user_id` to `null` or omit it entirely from the push. Do not pass empty strings.

4. **Deduplication**: If both Page View and History Change triggers fire simultaneously (e.g., on initial SPA load), implement logic to prevent duplicate `page_view` events for the same URL and timestamp.

5. **Timing**: Ensure all page context variables (`page_type`, `page_category`) are populated in the dataLayer **before** the event fires. Use GTM's dataLayer variable initialization or a DOM Ready trigger if values depend on page content.

6. **Fallback Values**: If `page_category` cannot be determined (e.g., on error pages), use a descriptive fallback like `"uncategorized"` or `"error"` rather than leaving it undefined.
