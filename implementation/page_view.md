# page_view

Tracks when a user views any page on the website, including initial page loads and single-page application (SPA) navigation.

---

## Overview

This event fires every time a user lands on a page or navigates to a new view within the application. It captures the type and category of page being viewed, along with the user's authentication state. This data is foundational for understanding traffic patterns, content engagement, and user journeys across the site.

---

## Trigger Rules

- **Must fire** when the browser completes an initial page load (DOM ready or window load, depending on implementation)
- **Must fire** when the URL changes via client-side routing in a single-page application (SPA) — e.g., History API `pushState` or `replaceState` events
- **Must NOT fire** multiple times for the same route change (deduplicate rapid-fire SPA navigation events)
- **Must NOT fire** on hash-only changes (`#anchor` links) unless they represent meaningful page views in the application's navigation model
- The `user_id` must be hashed or anonymized **before** the dataLayer push to comply with privacy regulations

## Business Rules & Edge Cases

1. **PII Compliance (Critical):** The `user_id` parameter is flagged as a PII risk. Before pushing to the dataLayer, hash the raw user ID using SHA-256 or another approved one-way hash. Never transmit plain-text email addresses, usernames, or database IDs.

2. **SPA Navigation:** For single-page applications, configure GTM with a History Change trigger in addition to the standard Page View trigger. This ensures route changes fire the event without a full page reload.

3. **Deduplication:** Implement logic to prevent duplicate fires during rapid navigation (e.g., user clicks multiple links quickly). A simple debounce of 100–200ms or a "last fired URL" check is recommended.

4. **Logged-Out Users:** When the user is not authenticated, set `user_login_status` to `"logged_out"` and either omit `user_id` entirely or set it to `null`. Do not use placeholder strings like `"anonymous"` or `"guest"`.

5. **Page Type & Category Values:** Use a controlled vocabulary for `page_type` and `page_category`. Coordinate with the analytics team to ensure consistent values across all pages (e.g., `"homepage"`, `"product_detail"`, `"cart"`, `"checkout"`, `"account"`).

6. **GTM Configuration:** Use the Page View trigger type for initial loads and the History Change trigger type for SPA navigation. Ensure both triggers populate the dataLayer with the same schema.
