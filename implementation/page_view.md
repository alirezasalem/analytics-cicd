# page_view

Tracks when a user views any page on the website, including initial page loads and single-page application (SPA) route changes.

---

## Overview

The `page_view` event fires every time a user lands on a page or navigates to a new view within the site. This is the foundational event for understanding user journeys, measuring traffic distribution across page types, and segmenting behavior by logged-in vs. anonymous users. It captures both traditional full page loads and client-side route changes in single-page applications.

---

## Trigger Rules

- **Must fire on:** Initial browser page load (document ready/DOMContentLoaded)
- **Must fire on:** Client-side route changes in SPA frameworks (React Router, Vue Router, etc.) when the URL or view changes without a full page reload
- **Must fire once per page view** — do not fire multiple times for the same URL without a navigation event
- **Must NOT fire on:** Partial DOM updates that don't represent a new "page" (e.g., modal opens, tab switches within the same view, AJAX content loads that don't change the route)
- **User authentication state** should be resolved before firing (wait for auth check to complete so `user_login_status` and `user_id` are accurate)

## Business Rules & Edge Cases

1. **SPA Navigation Handling:** In single-page applications, use GTM's History Change trigger (or equivalent framework lifecycle hooks) to detect route changes. Do not rely solely on the Page View trigger, which only fires on full page loads.

2. **PII Compliance (Critical):** The `user_id` parameter is flagged as a PII risk. Before pushing to the dataLayer, the user ID **must be hashed** (e.g., SHA-256) or anonymized. Never send raw email addresses, account numbers, or other personally identifiable information. This is required for GDPR/CCPA compliance.

3. **Authentication Timing:** If your site has async authentication (e.g., token validation on load), delay the `page_view` push until auth state is confirmed. Firing too early may result in incorrect `user_login_status` values.

4. **Logged-Out Users:** When the user is not authenticated, set `user_login_status` to `"logged_out"` and either omit `user_id` entirely or set it to an empty string. Do not send placeholder values like "guest" or "anonymous" in the `user_id` field.

5. **Deduplication:** Ensure the event fires exactly once per distinct page view. Guard against duplicate pushes from framework quirks (e.g., React's double-render in StrictMode during development).

6. **Page Type Consistency:** Maintain a controlled vocabulary for `page_type` values across the site. Document all valid values (e.g., "homepage", "product_listing", "product_detail", "cart", "checkout", "order_confirmation", "account", "blog_article", "error_404") to ensure consistent reporting.
