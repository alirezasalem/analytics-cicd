## Background

Product pages and listing pages currently lack standardised add-to-cart tracking. Without it we cannot measure cart-add rate by product, category, or traffic source, and we cannot fuel GA4 purchase-funnel reports or Google Ads remarketing audiences. This brief defines the single `add_to_cart` event that must fire whenever a user adds one or more items to the cart, regardless of the surface it happens on.


## Feature description

A user can trigger an add-to-cart action from multiple surfaces:

1. **Product Detail Page (PDP)** — clicks the primary "Add to Cart" button after optionally selecting a variant (size, colour, quantity).
2. **Product Listing Page (PLP) / Category page** — clicks an inline "Add to Cart" quick-action button on a product card.
3. **Recommendations / upsell widgets** — clicks "Add" inside a "You may also like" or cross-sell carousel.

In all cases the browser-side dataLayer push must happen only after the cart API returns a **success response (HTTP 2xx)**, not on button click, to avoid counting failed add attempts.


## Business questions

- What is the add-to-cart rate per product, category, and brand?
- Which traffic source / campaign drives the highest add-to-cart volume?
- What percentage of sessions that add to cart proceed to purchase (funnel drop-off)?
- Are variant selections (size, colour) correlated with add-to-cart abandonment?
- What is the average `value` (revenue potential) added per session?


## Events to track

### Event: Add to Cart

**When it fires:** On successful cart API response (HTTP 2xx) — not on button click. Fire once per add-to-cart action regardless of quantity.

**dataLayer push:**

```js
dataLayer.push({ ecommerce: null }); // Clear previous ecommerce object
dataLayer.push({
  event: "add_to_cart",
  ecommerce: {
    currency: "EUR",            // ISO 4217 — match transaction currency
    value: 29.99,               // price × quantity for items added this action
    items: [
      {
        item_id: "SKU_12345",           // required — variant-level SKU preferred
        item_name: "Classic Logo Tee",  // required — human-readable product name
        affiliation: "",                // optional — store or seller name
        coupon: "",                     // optional — coupon applied to this item
        discount: 0.00,                 // optional — monetary discount per unit
        index: 0,                       // optional — position in list (0-based)
        item_brand: "Acme",             // recommended
        item_category: "Apparel",       // recommended — top-level category
        item_category2: "T-Shirts",     // optional — sub-category
        item_category3: "",             // optional
        item_list_id: "pdp",            // recommended — surface id: pdp | plp | recs
        item_list_name: "Product Detail Page", // recommended — human label
        item_variant: "Black / M",      // recommended — selected variant
        location_id: "",                // optional — GA Place ID if relevant
        price: 29.99,                   // required — unit price after discount
        quantity: 1                     // required — units added this action
      }
    ]
  }
});
```

**Data to capture:**
- `currency` — ISO 4217 code (e.g. `EUR`, `USD`); must match the checkout currency
- `value` — `price × quantity` for all items added in this push; used in GA4 revenue reports
- `item_id` — variant-level SKU when available; fall back to product ID
- `item_name` — display name as shown to the user
- `item_brand` — brand name
- `item_category` / `item_category2` — GA4 supports up to 5 levels
- `item_variant` — concatenated selected options, e.g. `"Blue / XL"`
- `item_list_id` / `item_list_name` — surface where the add happened (`pdp`, `plp`, `recs`)
- `price` — unit price **after** any applied discount; do not send the original price here
- `discount` — monetary discount per unit (original price minus sale price)
- `quantity` — number of units added; must be ≥ 1
- `index` — 0-based position of the item in the originating list (for PLP/recs surfaces)
- `coupon` — coupon code applied at item level, if any

**Priority:** P0

**Edge cases:**
- **Multiple items in one add action** (e.g. bundle add) — include all items in the `items` array in a single push; do not fire one event per item.
- **Quantity update vs. fresh add** — if the user changes quantity on an existing cart line, treat the delta as a new `add_to_cart` (positive delta) or `remove_from_cart` (negative delta); do not fire `add_to_cart` for quantity=0.
- **PII** — do not include user name, email, or address in any ecommerce parameter.
- **SPA routing** — ensure `dataLayer.push({ ecommerce: null })` always precedes the event push to prevent bleed-over from a previous ecommerce event.
- **Duplicate prevention** — debounce rapid double-clicks; fire only once per successful API response using a per-request flag.
- **Failed API calls** — do not push the event if the cart API returns 4xx/5xx; surface the error to the user and suppress the dataLayer push.
- **Currency mismatch** — if the page currency differs from the user's billing currency, always use the displayed/transactional currency and document this in the implementation note.


## Scope

**In scope:**
- All surfaces where a user can add an item to the cart: PDP, PLP, recommendation widgets, quick-view modals
- Web (desktop + mobile browser)
- All markets / locales (currency field must reflect the locale)

**Out of scope:**
- Native mobile apps (separate brief required)
- Wishlist / save-for-later actions (not an add-to-cart intent)
- Back-in-stock notification sign-ups
- `remove_from_cart` event (separate brief)
- Server-side GTM forwarding configuration (covered in A6)


## Risks

| Risk | Notes |
|------|-------|
| Cart API response not accessible client-side | Coordinate with engineering to confirm API response is observable in the browser (XHR/fetch intercept or callback); if not, a custom JS event from the app layer is needed |
| Variant SKU unavailable at add time | Fall back to product ID; flag in spec as `NEEDS_CLARIFICATION` |
| Multiple add surfaces with different DOM structures | Each surface needs its own GTM trigger; consolidate in a single tag using a shared custom event |
| Currency inconsistency across markets | Define a single source-of-truth variable (e.g. a JS global or data attribute) that all implementations read |
| Duplicate fires on slow networks / retry logic | Implement a per-request boolean guard reset on each new user action |
