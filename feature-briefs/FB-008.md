## Background

The `purchase` event is the single most critical conversion metric for revenue reporting, ROAS calculation, and marketing attribution. Currently, we lack precision in tracking completed orders, which severely limits our ability to optimize paid acquisition campaigns and audit revenue accuracy against the backend order management system. Implementing this event securely ensures a definitive end to our e-commerce purchase funnel.


## Feature description

The purchase tracking will trigger at the final stage of the checkout conversion funnel.

1. **Transaction Processing:** The user submits their final order and payment information on the checkout page. The payment gateway processes and successfully confirms the transaction.
2. **Success Confirmation:** The application routes the user to the dedicated Order Confirmation ("Thank You") page.
3. **Data Extraction & Dispatch:** As the Order Confirmation page initializes, the frontend reads the transaction details (order ID, revenue totals, tax, shipping, and the product array) from the application state or secure backend payload and dispatches the data to the analytics platform.


## Business questions

- What is our total revenue, average order value (AOV), and purchase conversion rate over time?
- Which marketing channels and campaigns drive the highest return on ad spend (ROAS) and actual conversion volume?
- What are the most common product combinations purchased together in a single transaction?


## Events to track

### Event: purchase

**When it fires:** Immediately upon the successful loading of the Order Confirmation / Thank You page (DOM ready), or when the underlying order completion status is verified as successful. It must only fire if the transaction went through successfully.

**Data to capture:**
- `transaction_id`: The unique transaction/order confirmation reference string generated by the backend
- `value`: The total net revenue of the order (numeric value, excluding shipping and tax)
- `tax`: The total tax amount applied to the order
- `shipping`: The total shipping cost charged to the user
- `currency`: The active currency code (e.g., "EUR")
- `coupon`: The order-level discount/promo code used (pass null or empty string if none)
- `items`: An array of objects representing each purchased product, containing:
  - `product_id`
  - `product_name`
  - `price`
  - `quantity`
  - `variant`

**Priority:** P0

**Edge cases:** - **Page Refreshes & Back Button:** If a user bookmarks, reloads, or navigates back to the Order Confirmation page, the `purchase` event must **not** fire a second time. The implementation must include deduplication logic (e.g., checking a cookie or local storage registry of processed `transaction_id`s).
- **Asynchronous Payment Gateways:** For payment methods requiring external redirection (e.g., PayPal, Sofort), ensure the event captures correctly upon return to the confirmation route without dropping cart payload data.
- **Null/Zero Values:** Ensure currency values pass as pure numbers, never formatted strings containing symbols (e.g., pass `49.99`, do not pass `49,99 €`).

## Scope

**In scope:**
- All completed transactions originating from the web-based standard checkout flow on desktop, tablet, and mobile layouts.
- Transactions using credit cards, digital wallets, or localized invoice methods.

**Out of scope:**
- Subscriptions or recurring renewal billing events that happen offline without user action (handled by backend webhooks).
- Order cancellations, modifications, or refunds (to be handled in a separate `refund` tracking brief).


## Risks

| Risk | Notes |
|------|-------|
| Duplicate Transactions | Refreshing the thank-you page is the leading cause of artificially inflated revenue data. Deduplication based on the `transaction_id` at the container or storage level is mandatory. |
| Browser Disconnections | If a user closes the tab immediately after the payment processes but before the redirect to the confirmation page completes, the frontend event will fail to fire. A backup server-side tracking mechanism should eventually be evaluated to minimize data discrepancy. |
