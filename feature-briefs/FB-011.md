## Background

The creation of a new account is a milestone user acquisition metric. Tracking new registrations allows marketing teams to accurately compute cost-per-acquisition (CPA) for performance marketing campaigns and lets product teams analyze user onboarding flow activation. Differentiating new registrations from recurring logins is vital to maintain clean user lifecycle segments.


## Feature description

The tracking mechanism will capture successful new user registrations across all onboarding entries.

1. **User Action:** A new user completes the registration form (entering details like email and password) or clicks a single sign-on provider (SSO) on the registration screen.
2. **Account Creation Validation:** The backend database successfully registers the new account, returns a success confirmation, and generates a new internal user ID.
3. **Dispatch:** The application captures the account creation success state and dispatches the `sign_up` event containing the creation method before routing the user into their fresh onboarding experience or dashboard.


## Business questions

- Which marketing campaigns or referral sources drive the highest volume of new user account creations?
- What is the preferred signup method (e.g., Email vs. Google SSO) for our new users?
- What percentage of users who land on the registration page complete the signup process versus dropping off?


## Events to track

### Event: sign_up

**When it fires:** Immediately upon a successful account creation confirmation response from the backend (e.g., API 201 Created or specific auth provider success callback) and before any subsequent welcome redirects occur. It must *not* fire on validation failures (e.g., "Email already exists").

**Data to capture:**
- `user_id`: The newly generated unique, persistent database identifier for the user (must be a non-PII internal ID string)
- `method`: The registration provider used (e.g., "email", "google", "apple")

**Priority:** P0

**Edge cases:** - **SSO Login vs. SSO Signup:** If a user clicks "Sign up with Google" but already has an existing account linked to that email, the backend should route them as a login. In this scenario, fire a `login` event *instead* of a `sign_up` event.
- **Form Resubmissions:** If a user experiences slow network states and clicks the submit button multiple times, the tracking script must wait for the definitive API response and fire only a single `sign_up` event for the generated account.
- **Immediate Login Post-Signup:** Do not fire a standard `login` event immediately following a `sign_up` event within the same onboarding sequence, as account creation implicitly handles the initial session authentication.

---


## Scope

**In scope:**
- All successful new registration paths on the platform, including dedicated `/signup` pages, embedded checkout creation checkboxes, and modal registration overlays.
- Web responsive flows across desktop, tablet, and mobile browsers.

**Out of scope:**
- Profile updates or modifying registration fields after account creation.
- Native mobile app registration flows.


## Risks

| Risk | Notes |
|------|-------|
| PII Exposure | Data protection rules strictly prohibit sending any cleartext personal details (such as names or email addresses) in the event parameters. Only the raw backend tracking ID string and the method are allowed. |
| Mixed Login/Signup Counts | Relying purely on client-side button clicks to track signups will skew data with failed attempts and duplicate existing users. The hook must look directly at the database creation confirmation status. |
