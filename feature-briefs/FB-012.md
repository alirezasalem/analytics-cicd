## Background

The payment submission step is the final hurdle before a transaction is completed. High abandonment rates at this stage often point to usability issues, unexpected processing errors, or a lack of localized payment options. By tracking exactly when a user successfully saves or submits their payment details during the checkout flow, we can pinpoint payment-specific friction and optimize funnel conversion.


## Feature description

The event tracking will capture successful payment method submissions within the checkout application.

1. **User Action:** The user fills out their payment details (e.g., credit card info) or selects a digital wallet option on the checkout payment step and clicks the submission/save button.
2. **Payment Validation:** The payment gateway iframe, client SDK, or backend validation service verifies the payment details as valid/authorized.
3. **Dispatch:** The moment the application state updates to confirm the payment method has been successfully attached to the active order session, the event fires with the selected payment type and cart items array before moving to the final review or confirmation page.


## Business questions

- What is the drop-off rate between initiating checkout and successfully adding payment information?
- Which payment methods (e.g., Credit Card, PayPal, Invoice) are preferred by our high-value customers?
- Are users encountering a disproportionate number of payment-saving failures on mobile web compared to desktop web?


## Events to track

### Event: add_payment_info

**When it fires:** Immediately after the checkout application successfully validates and applies the user's payment details to the transaction session (e.g., following a successful payment token generation or gateway API 200 response). It must *not* fire on the raw submission click if payment authorization fails.

**Data to capture:**
- `payment_type`: The type/method of payment selected by the user (e.g., "credit_card", "paypal", "sofort", "applepay")
- `value`: The total monetary value of the cart being paid for (numeric value, excluding shipping/tax unless final)
- `currency`: The active currency code (e.g., "EUR")
- `coupon`: The coupon code applied to the order (pass null or empty string if none)
- `items`: An array of objects representing each product in the cart, containing:
  - `product_id`
  - `product_name`
  - `price`
  - `quantity`
  - `variant`

**Priority:** P0

**Edge cases:** - **Express Checkouts:** If a user chooses an express checkout option (like Apple Pay or PayPal Express) directly from the cart page, both `begin_checkout` and `add_payment_info` must be triggered in sequence as the checkout session initializes with payment information pre-filled.
- **Payment Method Updates:** If a user successfully adds a credit card, but encounters an issue on the subsequent order review page and changes their payment method to PayPal, a second `add_payment_info` event should fire capturing `payment_type: "paypal"`.
- **Declined Methods:** If the payment validation fails (e.g., invalid card number, expired date), do not fire the event.

---


## Scope

**In scope:**
- Successful submission or verification of payment details within the web responsive checkout funnel across desktop, tablet, and mobile views.
- Saved payment methods updated within a user's account settings page if it triggers a transaction authorization check (optional depending on system framework).

**Out of scope:**
- Capturing the actual payment instrument data (card numbers, CVVs, bank account routing details).
- Native mobile application checkout paths.


## Risks

| Risk | Notes |
|------|-------|
| PCI Compliance & Security | **Strictly Prohibited:** Under no circumstances should raw credit card numbers, CVV codes, expiration dates, or bank details be captured or transmitted as parameters. Only pass the generic `payment_type` string. |
| Asynchronous Gateway Redirects | If the payment type redirects the user off-site (e.g., standard PayPal redirect) before the event can be pushed on-site, tracking may be lost. The script must execute right before the redirect occurs or be queued to fire immediately upon successful return to the site. |
