## Background

The contact form is a vital lead generation and customer support entry point on our platform. Currently, we only track visits to the contact page, leaving us blind to how many users successfully complete and submit the form versus those who abandon it due to field complexity or validation confusion. Tracking successful submissions enables us to measure lead acquisition efficiency and optimize form UX.


## Feature description

The tracking mechanism will capture successful interactions with the contact or inquiry form.

1. **User Action:** The user fills out the required contact form fields (e.g., topic, inquiry details) and clicks the "Submit" or "Send Message" button.
2. **Form Validation:** The frontend completes client-side validation, and the backend API processes the submission, returning a successful confirmation status.
3. **Dispatch:** The moment the user interface renders the "Success / Thank You" message state (or receives the API 200/201 confirmation), the event fires with the form categorization metadata before any page reloads or redirects occur.


## Business questions

- What is the overall completion rate of the contact form, and which pages generate the most inquiries?
- What are the most common inquiry topics selected by users reaching out to support?
- Do users drop off more frequently on forms with optional fields versus strictly required fields?


## Events to track

### Event: generate_lead

**When it fires:** Immediately after the contact form submission is successfully validated and accepted by the backend API (e.g., upon an API 200 response or when the UI dynamically transitions to show the success message). It must *not* fire on the raw button click if validation fails.

**Data to capture:**
- `form_id`: The unique identifier of the form element (e.g., `main_contact_form`, `pdp_inquiry_form`)
- `form_name`: A descriptive name of the form context (e.g., `customer_support`, `b2b_sales_inquiry`)
- `lead_category`: The general topic or department selected by the user from the dropdown (e.g., `billing`, `technical_support`, `general_feedback`)

**Priority:** P1

**Edge cases:** - **Double Submissions:** If a user clicks the submit button multiple times while the API is loading, the tracking script must await the successful response and fire only a single `generate_lead` event.
- **Asynchronous Success Pages:** If the form redirects to a separate static `/thank-you-contact` page upon success, the event can be handled on that page load, provided the form metadata (`form_id`, `lead_category`) is passed along via state or session storage.
- **Validation Errors:** If the submission fails due to captcha issues, empty required fields, or server timeouts, do not trigger the event.

---


## Scope

**In scope:**
- Successful form submissions originating from the dedicated contact page, floating help desk widgets, or situational inquiry forms.
- Desktop, tablet, and mobile web responsive layouts.

**Out of scope:**
- Tracking individual field focus, keystrokes, or validation errors (e.g., "invalid email format") within this specific brief.
- Newsletter signups (covered by a separate subscription tracking brief).


## Risks

| Risk | Notes |
|------|-------|
| PII Exposure | **Strictly Prohibited:** Under no circumstances should the user's name, cleartext email address, phone number, or the free-text message body be captured or transmitted as parameters. Only abstract structural parameters (`form_id`, `lead_category`) are allowed. |
| False Positives via Button Clicks | Binding the tracking trigger strictly to the click event of the submit button will capture failed validation attempts and severely skew lead generation data. |
