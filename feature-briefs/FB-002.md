## Background

Understanding user purchase intent begins when a user adds an item to their cart. Currently, we cannot measure the conversion rate from Product Detail Pages (PDP) to cart additions. Capturing this event is essential for evaluating product popularity, optimizing PDP layouts, identifying high-intent users, and building retargeting funnels for abandoned carts.


## Feature description

The event tracking will monitor user interactions on all Product Detail Pages (PDP).

1. **User Action:** A user visits a PDP, selects any necessary product variations (such as size, color, or quantity), and clicks the "Add to Cart" button.
2. **System Validation:** The application processes the request. Once the backend or local cart state successfully confirms that the item has been added to the user's cart, the event payload is generated.
3. **Dispatch:** The event fires seamlessly in the background, sending the specific product details and metadata to the analytics platform without interrupting the user's shopping experience.


## Business questions

- What is the overall PDP-to-Add-to-Cart conversion rate across the site?
- Which specific products or product categories are added to the cart most frequently but experience high abandonment rates later?
- Does changing product variants (e.g., color, size) or viewing specific images impact the likelihood of a user adding an item to the cart?


## Events to track

### Event: add_to_cart

**When it fires:** Immediately after the cart state successfully updates following a user clicking the "Add to Cart" button on a Product Detail Page (e.g., upon a successful API 200 validation response or local state confirmation). It must *not* fire on the raw button click if the item fails to add.

**Data to capture:**
- `product_id`: The unique identifier/SKU of the added item
- `product_name`: The plain text name of the product
- `price`: The unit price of the item (as a numeric value)
- `quantity`: The number of items added to the cart in this action
- `product_category`: The primary category/hierarchy of the product
- `variant`: The selected option combination (e.g., "Size: M, Color: Black")
- `currency`: The active currency code (e.g., "EUR")

**Priority:** P0

**Edge cases:** - **Rapid Clicks:** If a user clicks the button multiple times quickly before the application responds, ensure the tracking logic waits for the successful state validation and only dispatches individual events per unique successful item addition.
- **Out of Stock/Errors:** If the API returns an error (e.g., stock unavailable), do not fire the `add_to_cart` event. 
- **Quantity Adjustments:** If a user increases the quantity from "1" to "3" directly on the PDP before clicking the button, ensure the event captures `quantity: 3` instead of firing three separate events.

## Scope

**In scope:**
- All standard Product Detail Pages (PDP) across desktop, tablet, and mobile web views.
- Success-validated item additions to the shopping cart originating from these pages.

**Out of scope:**
- Quantity updates, removals, or additions that occur *inside* the mini-cart drawer or on the dedicated Cart/Checkout pages (these will be covered in separate briefs).
- Native mobile app tracking.


## Risks

| Risk | Notes |
|------|-------|
| Optimistic UI vs. API Failures | If the UI instantly shows "Added" before the API confirms success, firing the event early will skew data if the API ultimately fails. Tracking must bind to actual cart success states. |
| Inaccurate Currency/Pricing | Multi-currency sites risk passing the wrong currency symbol or string formatted numbers. Pricing must be passed as a clean floating-point number alongside a ISO currency code. |
