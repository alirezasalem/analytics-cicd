## Background

Understanding user authentication state is foundational to mapping behavioral profiles over time. Currently, we struggle to stitch anonymous guest sessions with post-login user accounts accurately. Tracking successful login events ensures we can understand how users interact with the platform across multiple sessions and devices, while also letting us evaluate the performance and adoption of different authentication options (e.g., Google OAuth vs. Email/Password).


## Feature description

The tracking mechanism will capture successful user authentication across all entry points (Login page, checkout login walls, and popups).

1. **User Action:** The user inputs their credentials or selects a single sign-on provider (SSO) and submits the login form.
2. **Authentication Validation:** The identity provider or backend auth service authenticates the user credentials successfully.
3. **Dispatch:** As soon as the application receives the successful authentication response and stores the session token, the login event fires with the associated user metadata before the user is redirected.


## Business questions

- What percentage of our daily active users interact with the site while logged in versus logged out?
- Which login method has the highest success/adoption rate among our users?
- Are users who log in using SSO (e.g., Google, Apple) exhibiting higher long-term retention compared to those using standard email logins?


## Events to track

### Event: login

**When it fires:** Immediately upon a successful authentication response (e.g., API 200 containing the JWT or session token) and before the application executes a redirect to the home or target dashboard page. It must *not* fire if the authentication fails due to incorrect passwords or server errors.

**Data to capture:**
- `user_id`: The unique, persistent database identifier assigned to the authenticated user (must be hashed or a non-PII internal ID)
- `method`: The authentication provider used to log in (e.g., "email", "google", "apple", "facebook")

**Priority:** P0

**Edge cases:** - **Persistent Logins ("Remember Me"):** This event should only fire when a user actively inputs credentials/clicks to authenticate. If a user returns to the site and is automatically logged in via a valid, pre-existing refresh token or session cookie, do *not* fire a new `login` event (re-authentication is typically handled by setting the user state on the initial `page_view`).
- **Checkout Login Redirects:** If a user logs in mid-funnel during checkout, ensure the login event fires successfully without interrupting the checkout redirect state or losing the cart context.
- **Immediate Redirects:** Ensure the event is dispatched or successfully queued in local storage before the browser page reloads entirely during traditional server-side redirect patterns.

---


## Scope

**In scope:**
- Successful authentications on the dedicated `/login` page, modal login popups, and checkout-step authentication walls.
- Desktop, tablet, and mobile web responsive environments.

**Out of scope:**
- Brand new user registrations (this should be tracked via a separate `sign_up` event brief).
- Failed login attempts due to validation errors (e.g., "Password incorrect").
- Native mobile app logins.


## Risks

| Risk | Notes |
|------|-------|
| PII Leakage | Never pass cleartext emails, usernames, or passwords into the event parameters. Only the plain database user string (or hashed representation) and the authentication method string are permitted. |
| Race Conditions during Redirects | If the application triggers a hard window location reload immediately upon login success, the tracking script may be destroyed before the network request finishes dispatching. Use transport beaconing (`keepalive: true`) or queue the event. |
