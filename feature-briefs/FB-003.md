## Background

The checkout funnel is the most critical phase of the user journey for conversion optimization. While we can track cart additions, we currently have a blind spot regarding how many users actively initiate the checkout process versus those who abandon their carts passively. Tracking the exact moment a user transitions from shopping to checking out allows us to isolate friction points at the very start of the purchase funnel.


## Feature description

The tracking mechanism will capture the transition from the shopping/cart state to the formal checkout application environment.

1. **User Action:** The user navigates to the cart page or opens the mini-cart drawer and clicks the "Proceed to Checkout" button (or "Checkout as Guest" / "Express Checkout" buttons).
2. **Funnel Initialization:** The web application processes the request and initiates the checkout route/view loading state.
3. **Dispatch:** Once the first step of the checkout process successfully initializes (e.g., the contact information or shipping form renders), the event dispatches the entire contents of the cart as an array, alongside transaction metadata, to the analytics server.


## Business questions

- What percentage of users who add items to their cart actually begin the checkout process?
- Is there a higher abandonment rate at the start of checkout for users with higher cart values versus lower cart values?
- How does the utilization of express checkout buttons (e.g., Apple Pay, PayPal) at the initiation stage impact overall funnel completion compared to standard checkout?


## Events to track

### Event: begin_checkout

**When it fires:** Immediately upon the successful loading of the first step of the checkout interface (e.g., on SPA route resolution of the checkout entry page, or upon a successful API 200 initialization response if checkout is heavily server-driven). It should fire once per checkout session initialization.

**Data to capture:**
- `value`: The total monetary value of all items in the cart combined (numeric, excluding shipping/tax unless pre-calculated)
- `currency`: The active currency code (e.g., "EUR")
- `coupon`: The cart-level discount code applied (pass null or empty string if none)
- `checkout_option`: The method used to enter checkout if distinguishable (e.g., "standard", "express_paypal", "express_applepay")
- `items`: An array of objects representing each product in the cart, containing:
  - `product_id`
  - `product_name`
  - `price`
  - `quantity`
  - `variant`

**Priority:** P0

**Edge cases:** - **Guest vs. Logged-in Re-routing:** If an unauthenticated user clicks checkout, is redirected to a login/register wall, and then successfully lands on the checkout page, ensure the event fires *only once* when they finally reach the checkout step, not before the login wall.
- **Cart Alterations:** If a user leaves the checkout page to add more items to their cart and then returns to the checkout, a new `begin_checkout` event should fire reflecting the updated cart value and items array.
- **Empty Carts:** Prevent the event from firing entirely if the cart object is empty due to technical errors or direct URL manipulation to the checkout route.

---


## Scope

**In scope:**
- Checkout initiations originating from the full cart page, mini-cart drawers, or direct product express-checkout overrides.
- All web-based desktop, tablet, and mobile responsive flows.

**Out of scope:**
- Intermediate funnel steps (e.g., entering shipping address, selecting payment methods), which will be tracked via separate step-specific tracking briefs.
- Native mobile application checkout paths.


## Risks

| Risk | Notes |
|------|-------|
| Array Formatting Errors | The analytics endpoint expects a strictly structured array of nested item objects. If any product metadata is missing fields or incorrectly typed (e.g., string instead of number for price), the entire payload may drop. |
| Login-Wall Drops | Skewed drop-off data can occur if the login/register intermediary page is mistaken for the checkout entry point itself. The trigger must explicitly bind to the actual checkout step initialization. |
